# Code Promotion : What and How


You must promote a combination of the compiled artifact, 
environment-specific configuration, 
and database schema changes to ensure system stability.

```plaintext
[ Build Stage ] ──> Creates Immutable Artifact (e.g., Image v1.2)
                        │
                        ▼
[ Staging Env ] ──> Combines Image v1.2 + Staging Config + Schema Migrations
                        │  (Test & Validate)
                        ▼
[ Prod Env ]    ──> Combines Image v1.2 + Prod Config + Schema Migrations
```

## Purpose

Per environment:

```plaintext
[ Local/Cloud Dev ] ──> [ Staging / Pre-Prod ] ──> [ Production ]
(Write & Unit Test)     (Simulate & Integrate)     (Live User Traffic)
```

## What to Promote

* **Immutable Artifacts**:   
    This includes Docker images, compiled binaries, or zip packages. Build them once in the build stage and use the exact same file across all environments.
* **Environment Configuration**:   
    Key-value pairs, feature flags, and secrets (like API keys) unique to each environment.
* **Database Migrations**:   
    Schema changes or data migration scripts required by the new artifact version.

## How to Promote (Step-by-Step)

The Dev environment creates the code that triggers the build.
The build happens after Dev is finished, not before.

## The Exact Order of Events

The **Build Stage** happens right *after Dev and before Staging*.

The build stage does not happen inside the Dev environment itself. 
Instead, it happens in a neutral zone—your CI/CD tool 
(like GitHub Actions, GitLab CI, or Jenkins)—immediately 
after a developer finishes their work in Dev and pushes their code.

```plaintext
1. DEV ENV  ──> Developer writes code, tests locally, and pushes to Git.
                     │
                     ▼
2. BUILD    ──> CI/CD tool compiles the pushed code and creates the image.
                     │
                     ▼
3. STAGING  ──> The newly built image is deployed here for QA testing.
                     │
                     ▼
4. PROD     ──> The exact same image is promoted to Live Users.
```

## Why It Flows This Way

1. Dev is the birthplace: The Dev environment is where the raw source code lives and changes. You cannot build an image until the developer finishes writing the code in Dev.
2. The Build is the exit gate: Once the developer is happy, they push their code to a repository (like GitHub). This push triggers the Build Stage.
3. The Image moves forward only: Once the build creates the image, that image only moves forward into Staging and Production. It never goes backward into the Dev environment.

## The Only Exception (Local Testing)

The only time an image goes into Dev is during local development. 
A developer might build a temporary image on their own laptop just to make sure it runs before pushing their code to Git. However, that temporary local image is discarded. 
The official build that goes to Staging and Prod is always generated by your central CI/CD pipeline.

## The Lifecycle Pipeline

```plaintext
[ Dev Environment ] 
       │  (Developer writes code & runs local tests)
       ▼
 ── PUSH CODE ──
       │
[ Build Stage (CI/CD) ] ──> Creates the Immutable Artifact (e.g., Docker Image)
       │
       ▼
[ Staging Environment ] ──> Deploys the artifact + Staging Config (QA Tests)
       │
       ▼
[ Prod Environment ]    ──> Deploys the *same* artifact + Prod Config (Live Users)
```

## Why Build Happens Exactly Here

* Dev is the source: Code only changes in Dev. Once a developer completes a feature, the code freeze for that version occurs.
* Build once, deploy many: The Build Stage turns that static code into a permanent artifact (like a Docker image).
* Staging and Prod only consume: Staging and Production never compile code or run builds. They simply download the finished artifact created right after the Dev phase and run it.

## Summary

1. **Build Once**: Compile your code and package your artifact during the initial build phase. Label it with a unique version or commit SHA.
2. **Deploy to Staging**:
    * Pull the immutable artifact.
    * Inject the Staging configuration via environment variables.
    * Apply database migrations against the Staging database.
3. **Test**: Run integration and automated tests in Staging to validate the release.
4. **Promote to Production**:
    * Pull the exact same immutable artifact used in Staging.
    * Inject the Production configuration variables.
    * Apply database migrations against the Production database.
5. Release: Route live user traffic to the newly deployed containers.

# 12-Factor using Trunk-based Branching

A single environment variable declares the target environment. 
You change that one variable, and your entire application adapts downstream.
In DevOps, **this variable is universally called** `NODE_ENV`, `APP_ENV`, or `ENVIRONMENT`.

## The Downstream Ripple Effect

When your CI/CD pipeline sets `ENVIRONMENT=staging` or `ENVIRONMENT=production`, 
it triggers a chain reaction across your app's architecture:

```plaintext
                  ┌──► 1. Config Loading ──► Pulls correct DB credentials & API keys
                  │
[ ENVIRONMENT ] ──┼──► 2. Feature Flags  ──► Enables experimental features only in Staging
                  │
                  └──► 3. Log Levels     ──► Switches to "debug" in Staging, "error" in Prod
```

1. **Config Loading**: Your code detects the variable and pulls the correct matching resources (e.g., if staging, connect to the staging database; if production, connect to the production database).
2. **Feature Flags**: External flag systems (like LaunchDarkly or Flipt) use this variable to serve different feature sets to different environments [12factor.net].
3. **Log Levels**: Your logging framework reads this variable to decide how chatty to be (e.g., verbose DEBUG logs for Staging, but quiet, cost-saving ERROR logs for Production).

## What This Looks Like in Code

Your application code never changes between environments. 
It simply queries the system environment variable at startup:

```python
import os

# The code is identical everywhere; only the system value changes
current_env = os.getenv("APP_ENV", "development")

if current_env == "production":
    # Enable strict security, high caching, and production payment gateways
    configure_production_settings()
else:
    # Enable detailed error pages and mock payment gateways
    configure_testing_settings()
```

## The 12-Factor Rule to Remember

According to the 12-Factor App methodology you looked at earlier, the code must never hardcode these switches [12factor.net]. The value must be injected into the container by the hosting platform (like AWS, Kubernetes, or Render) at the exact moment the container spins up.

In a modern Trunk-Based Development setup, CI and CD are almost always treated as two completely separate tools or distinct, decoupled pipelines [1].
While some software suites (like GitHub Actions or GitLab CI) can do both, they use different mechanisms to execute them.

## Why They Are Separated

```plaintext
[ Developer Commits to Trunk ]
               │
               ▼
┌──────────────────────────────┐
│       1. CI TOOL             │  ◄── Event-Driven (Triggered by Code)
│  (Test, Compile, Build Image)│
└──────────────┬───────────────┘
               │
               ▼  [ Pushes Immutable Artifact ]
┌──────────────────────────────┐
│    2. ARTIFACT REGISTRY      │
└──────────────┬───────────────┘
               │
               ▼  [ Reads TARGET_BUILD_VERSION ]
┌──────────────────────────────┐
│       3. CD TOOL             │  ◄── State-Driven (Triggered by Ops/Config)
│  (Deploy to Staging/Prod)    │
└──────────────────────────────┘
```

## 1. The CI Tool (Continuous Integration)

* The Job: Focuses on code validation. It runs tests, checks linting, compiles the code, and builds the immutable artifact (e.g., a Docker image) [1, 2].
* The Trigger: It is strictly event-driven by Git. It triggers automatically every time a PR is opened or code is merged into the trunk.
* The Lifecycle: Short-lived. It starts when code changes, runs for a few minutes, saves the build artifact to a registry, and shuts down [1].

## 2. The CD Tool (Continuous Delivery / Deployment)

* The Job: Focuses on environment state. It takes the artifact built by the CI tool and configures your cloud infrastructure to run it [1, 2].
* The Trigger: It is driven by configuration or command. It triggers when you update your TARGET_BUILD_VERSION variable, push to a configuration repo (GitOps), or click a "Promote" button.
* The Lifecycle: Long-lived and continuous. Tools like ArgoCD, Flux, or AWS CodeDeploy constantly monitor your environments to ensure the live application matches your target version variable [2].

## Common Tooling Examples

* CI Only: Jenkins, CircleCI, TravisCI.
* CD Only: ArgoCD, Flux, Octopus Deploy, Spinnaker [2].
* All-in-One Suites (Separated via Configuration): GitHub Actions (using standard workflows for CI, and GitOps or Environments for CD) [1].

To help you map out your architecture, are you leaning toward an all-in-one platform like GitHub/GitLab, or are you looking to pair a dedicated CI tool with a GitOps CD tool like ArgoCD?
